1. Dont overclock
2. Backoff cache and dram wait states.
3. disable internal/external cache (if this solves it, you know you're 
running your hardware too hard)
4. Get a new motherboard/ram/cache thats reliable.

This is indeed a hardwareproblem.

This occures when you have

  - defect mainboard
  - wrong kind of memory for your mainboard
  - defect memory
  - defect external cache
  - defect prozessor (especially bad nternal cache)
  - wrong bios setup
  - ...

For short everything having to do with the memory system can cause
this error.

I had the same problem and spend half a year to fix it.

Now everything is OK. I have had a defect mainboard together with
defect external cache AND defect memory.

I have seen a defect mainboard problem too... but before this, I have seen
something more weird:

An ESDI disk controller would randomly read bad data when an ethernet card
was inserted into the system (several different cards with same result,
and they work fine in other machines or with an ISA controller).

Since executables are paged from disk (on the machine in question, there
was also swapping going on, so even more chance to read somethig bad),
you can get SEGV by reading bad data too.

What made this really weird was that the gcc crashes were about 95%
reproducible - so it looked like a software problem first...

Well, in general, every part of the mass storage system and everything that
can affect it - that includes every card on the mainboard, the power supply
and external electromagnetic interference - could be a cause for the crashes
too.

Thus, you kan never tell the problem till you fix it. And even then, the
fix might be to replace a perfectly functioning part.

Linux malloc() will indeed fail if, at the time of the call, there is
not enough free VM to satisfy the request.  This breaks down rather
badly when programs calculate their memory requirements and then
allocate a large amount of memory in multiple chunks without using it.
This is not the way that 'gcc' typically allocates memory.  It
normally allocates a block of memory and then uses it before
allocating another, and the underlying sbrk() system call works as
expected in that case.

frodef@stud.cs.uit.no (Frode Vatvedt Fjeld) writes:
>On older (2.6.3) versions of gcc I have had SEGV's caused by gcc
>not checking a failed malloc. In cases where the crash is reproducible,
>I'd rank this as the most probable cause.

I don't see how this can be, since unless you've set limits on your
process's virtual memory size with limit or ulimit, malloc won't fail
on Linux (except for ridiculously large sizes).  That is, it returns
success, and the pages belong to you, but no backing store is allocated
until you dirty the pages (meaning there may be a problem at a later
point).  But you don't get a SEGV from this optimistic allocation, since
the system goes through gyrations to try to find a page and starts killing
processes with SIGKILL if it can't.
