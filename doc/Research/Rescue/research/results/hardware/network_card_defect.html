My first guess is that some of your hardware is bad. You might be able
to detect this by examining the output of "ifconfig eth0" on the Linux
box. This includes a listing of the number of errors, dropped, and
overrun packets, for both incoming and outgoing directions. If you see
large numbers of errors, drops, or overruns, it could be bad hardware.
(Note that "large" is a relative term, but certainly anything over 1% or
so is excessive.) Don't worry about collisions, unless they're over 10%
or so.

To try to further isolate this, I'd suggest buying a crossover cable and
connecting your computers with it, rather than via your current two
regular cables and hub. If the problem goes away, then you know it's due
to the hub or one of your existing cables. If not, then it's either not
a hardware problem or it's one of the network cards.

It's also possible that futzing with the TCP window or MTU size in
Windows would improve matters, but these are basically performance
issues; even if set incorrectly, they shouldn't cause your file
transfers to hang outright.

Well, looks like tcp.ip is running fine.  Perhaps a NetBIOS issue ?  If you
have these problems using ftp, I assume you are using IP addresses and not
NetBIOS names ?  If that is the case, then NetBIOS is not the culprit.

I would stick to testing with ftp, since Samba relies upon NetBIOS and it
would be good to ascertain whether IP access is working fine, before moving
to Samba.

Any IP conflicts, like two DHCP servers (router and Linux ?)  Any issues
from the ifconfig output on Linux ?

Hi Michael,

  Your problem regarding your network not being reachable is as a result of
you not specifying a gateway address on your windows machine.  You need to
specify the IP address of the Linux adapter on your Windows box, as the
machine doesn't know how to get packets to the linux box without a gateway.

  Am I right though, are you trying to do masquerading with just the one
