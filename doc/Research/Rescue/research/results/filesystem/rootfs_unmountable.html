Rootfilesystem cannot be mounted

Deutsch

Support knowledgebase (maddin_rootfs)

Symptom:

When trying to boot your S.u.S.E. Linux system, the booting process stops with a message like 
Kernel Panic: VFS: couldn't mount root
It is important to know, when this occurs:

  1. During the installation

  2. During the installation, at the end of the booting process

  3. In the running system

  4. After the creation of a new kernel
The following list of possible causes will refer to these topics. There will also be hints on articles that might provide a possible solution in these situations.

  • (1/2/3/4) You have a SCSI system. The root partition is located on a SCSI drive. The SCSI controller isn't recognized by the running kernel.

    This is the likely cause if the kernel displays a message like the following at boot-time : 

    scsi: 0 hosts

    Solution :

    Adaptec 2940 bleibt nach Installation hängen
    Adaptec 2940 mit DOS und loadlin
    Adaptec 2940 nicht erkannt
    SCSI-Adapter auf SoundBlaster
    Ultrastore 34F wird nicht erkannt

  • (1/2) When booting from CD during the installation the specified CD-ROM drive isn't recognized. The previous topic might also apply, if it is a SCSI CD-ROM. When loading the Rootimage from CD the latter isn't recognized and the system hangs.

    Solution :

    Make sure you entered the necessary kernel parameters correctly (if these are needed). Especially the values for IRQ and port adress are important.

    For further information see

    CDROM: CDU31 A Sony
    CDROM: Installation von CD-ROMs am Parallelport
    CDROM: ältere Mitsumi-Laufwerke (FX001-D)
    Media Vision Pro Sonic 16: Probleme mit CDROM-Interface

  • (3/4) The bootmanager LILO wasn't configured correctly. The root-partition specified in the file /etc/lilo.conf isn't correct.

    Solution :

    Boot the system with the bootdisk that comes with the S.u.S.E. Linux distribution or - even better - the bootdisk that was created during the installation. In the first case, after having loaded the necessary modules, choose the menu item Boot installed system. This way you can boot from your root partition. If this fails, please read the articles

    Booten von der S.u.S.E. Boot-Diskette.
    Notfalldiskette erstellen

    Once you're in the system, change the file /etc/lilo.conf such that it contains the correct root partition. Enter the command lilo and reboot the system in order to test the configuration.

    For further information see the article Linux über 1024 Zylinder

  • (3/4) When starting Linux from DOS using the Linux loader loadlin (resp. linux.bat), a wrong parameter for the root partition is specified. It is also possible, that there is a wrong root partition in the file c:\loadlin\linux.par .

    Solution :

    Please only specify the device-name WITHOUT leading /dev/ in LINUX.BAT, e.g. 

    c:\loadlin> linux hda5

    Also the following article might contain some important information : Adaptec 2940 mit DOS und loadlin

  • (4) During the reconfiguration and recompilation of the kernel no support for SCSI Low-Level drivers and/or SCSI discs was chosen or it was only activated as a module.

    Solution :

    In SCSI systems both the SCSI disk support and the low level drivers must be compiled directly into the kernel and may not be compiled as a module.

    Therefore : Boot the system with the bootdisk that comes with S.u.S.E. Linux or - even better - the bootdisk that was created during the installation. In the first case, after having loaded the necessary modules, choose the menu item Boot installed system. This way you can boot from your root partition.

    Now reconfigure, compile and install the kernel (see manual).

  • (4) During the reconfiguration and recompilation of the kernel no support for (E)IDE and/or EIDE drives was chosen.

    Solution :

    Please procede like in the previous topic and choose support for EIDE devices.

  • (4) During the reconfiguration and recompilation of the kernel no support for the ELF binary format was chosen or this support was only activated as a module.

    Solution :

    Boot the system with the bootdisk that comes with S.u.S.E. Linux or - even better - the bootdisk that was created during the installation. In the first case, after having loaded the necessary modules, choose the menu item Boot installed system. This way you can boot from your root partition.

    Compile and install a kernel, that has support for the ELF binary format. This support may NOT be activated as a module !

  • (4) During the reconfiguration and recompilation of the kernel no support for the EXT2 filesystem was chosen or this support was only activated as a module.

    Solution :

    Boot the system with the bootdisk that comes with S.u.S.E. Linux or - even better - the bootdisk that was created during the installation. In the first case, after having loaded the necessary modules, choose the menu item Boot installed system. This way you can boot from your root partition.

    Compile and install a kernel, that has support for the EXT2 filesystem. This support may NOT be activated as a module !

  • (3/4) After the recompilation of the kernel /etc/lilo.conf wasn't updated or lilo wasn't executed. As a result the correct kernel won't be started or the wrong root partition is being used.

    Solution :

    Whenever you copy a new kernel to the Root-directory /, lilo MUST be executed. Otherwise it won't find the new kernel. If the new kernel has a name, that is not yet listed in /etc/lilo.conf, you will also have to adapt this file.

    Boot the system with the bootdisk that comes with S.u.S.E. Linux or - even better - the bootdisk that was created during the installation. In the first case, after having loaded the necessary modules, choose the menu item Boot installed system. This way you can boot from your root partition.

    Make sure that /etc/lilo.conf fits your needs, then execute lilo.

  • (1/2/3/4) There is a failure of the hardware of your SCSI controller, the EIDE controller or the harddisk (This is unlikely, though).

    An indication for such failures are frequent read-/write-errors or defective boot sectors.

    Solution :

    If one of the devices is really broken, the only solution is to replace it.
