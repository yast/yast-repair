From the error messages I conclude, that the superblock itself seems to be
OK, but the blocks with the block descriptors contain invalid data. The
"group 0" is symptomatic: the next block after the superblock itself is
probably corrupted.

First a comment: get the 0.5b version of the e2fsck utilities. Parts of the
e2fsck code were corrected, also related to the block bitmap in the groups.

Then, the "-r" option would be not needed anymore.

OK, and more constructive suggestions:
   1. try an alternate superblock (it has its own copy of the group
      descriptors)
	e2fsck -b 8193 /dev/hda3

      ... or another alternate superblock at the block number 1+i*8192

   2. When all inspected superblocks still generate problems, one can
      go to more radical actions: "mke2fs -S" can create a new group
      descriptors.
      This is, however, quite drastic repair: you would not loose any
      additional files or directories by this, but you may loose the most
      of the names (the subsequent e2fsck will create a lot of entries
      in the /lost+found)

Another good approach would be to recompile the e2fsck with the "-g" flag
and to go with the "gdb" debugger to see how the rest of the data in the
superblock area looks like and what is the extent of the damage.
You can send me the samples of your superblocks and I can investigate them
using the tools which I have developed examining similar cases in the past.

Send me e-mail if you need more assistance.

	Zenon
