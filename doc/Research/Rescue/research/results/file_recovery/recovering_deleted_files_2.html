No, sorry. Because of the way linux stores stuff on the disk, and
because that disk is heavily buffered, there is no way for you to
'undelete' a file.

However if you are paranoid about deleting things by accident in the
future, you could always write a little script calle 'del' (or
something) that mv's stuff to a trash directory, where you could go and
examine your rubbish before deleting it for real.

I'm beta'ing a c-program called nrm which replaces (or aliases)
/usr/bin/rm(1).  Nrm attempts to be script-compatible with the
behavior and return codes of rm(1).

The basic idea is to move the deleted file into a hidden subdirectory
".gone".  So deleting "/home/walker/papers/myreport.txt" will create the
file "/home/walker/papers/.gone/myreport.txt".

This makes the file "dissappear" unless you give /bin/ls the "-a" option.

A nightly cron job deletes all .gone files that are older than (say)
three days.

The "-s" option to nrm will append a timestamp to the file name to ensure
that deleted files with the same name will not collide.  This is nice
to use with text files or databases to allow backing-out speculative
changes.

Email me if you want a source sharfile.

In fact, there _is_ a way to undelete files, if you are quick enough.
Midnight commander has so-called "undelete file system"
Just type (in mc) "cd undel://dev/hda1" (or what is your partition)
Then sort inodes by time, find (using F3) your files (yes, it is
somewhat difficult) and copy them to OTHER partition.

But, this way you can restore only files <=12kbytes.  It seems that
linux physically removes disk block nubmers it the first indirect block.

Anyway, you can "dd" the whole disk (or it's parts) to a file in other
partition and find parts of your text there.  I've done that once :-)

OK, all this is a bit after the fact, but here are some tips:

1) Backup, backup, backup
2) chmod ugo-w important-directory
3) chmod ugo-w important-files
4) do NOT alias rm to 'rm -i': you may come to rely on it, which is worse in
   case you find yourself in a different situtation. For this reason, I
   would also hesitate to alias rm to commands that shift a file to a trash dir
5) nice old trick: create an (empty) file of the name '-i' in your
   directory (e.g. as "touch ./-i"; delete with rm ./-i). When doing rm *,
   the shell will (usually, i.e. if wildcard expansions aka globbing is
   enabled) expand this into 
      rm -i file.c README a.out file.o
   So the -i will then be interpreted as a command line switch to rm, rather
   than a file name. 
6) If you use emacs for editing, look into the various versioning options. It
   can be very clever, e.g. it may automtically leave always the 3 oldest and
   5 youngest file versions around 
7) Use RCS (revision control system; see also CVS) This will also allow you
   to go back to previous versions of a file. Make sure the *,v files are in
   a subdir RCS, otherwise you may still get hit by a rm *. It is quite space
   efficient (more so than copying files); I once had > 60 fairly different
   versions of a file, the total of which requiring only twice the average
   size of the typical version (this was a text file, not a binary).


Hope this helps,
